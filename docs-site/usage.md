# Usage

This page focuses on the practical behavior of the two commands shipped by the
package.

## `clamdscan-progress`

### Default targets

If you run the command without paths, it uses `CTS_DEFAULT_TARGETS` from
`/etc/clamdscan-tools/clamdscan-tools.conf`.

```bash
sudo clamdscan-progress
```

By default this is intended for common interactive scanning, not necessarily a
full system sweep.

### System targets

Use `--system` to switch from default targets to `CTS_SYSTEM_TARGETS`.

```bash
sudo clamdscan-progress --system
```

This is the mode to use when you want a broader scan over paths such as
`/home`, `/etc`, `/var` or `/opt`, depending on your config.

### User-only scan

Without `sudo`, the tool scans only what the current user can read.

```bash
clamdscan-progress ~/Descargas ~/Documentos
```

In this mode:

- no attempt is made to start `clamav-daemon`
- unreadable files are naturally skipped by the OS permission model
- runtime files may fall back to `~/.local/state/clamdscan-tools/` and
  `~/.local/share/clamdscan-tools/` if the configured system paths are not writable

### Resume an interrupted scan

Each run writes a state file and a file list. To resume:

```bash
sudo clamdscan-progress --resume /var/lib/clamdscan-tools/state/scan_TIMESTAMP.state
```

Resume is useful when:

- the scan was interrupted with `Ctrl+C`
- the machine rebooted
- you intentionally stop a large scan and continue later

Important limitation:

- resume does not currently detect if a file changed after it was marked done

### Dry run

Build the file list and write progress state without invoking `clamdscan`:

```bash
clamdscan-progress --dry-run ~/Descargas
```

This is useful for validating exclusions and target selection before a real
scan.

## `clamdscan-watch`

Show the latest log:

```bash
clamdscan-watch
```

Follow the active log like `tail -f`:

```bash
clamdscan-watch --follow
```

Inspect a specific log file:

```bash
clamdscan-watch --log /var/log/clamdscan-tools/scan_TIMESTAMP.log
```

## Runtime files generated by a scan

Each run may produce:

- a main log file
- a state file for resume
- a `.files` list generated from `find`
- a `.find.err` file containing `find` stderr only

That last point matters: `find.err` is not where `clamdscan` file scan failures
are stored. Scan failures are written inside the main log.

## Root vs non-root behavior

### As root

- uses `clamdscan --fdpass`
- can scan protected paths
- can move infected files to quarantine
- can attempt to start the configured `clamav-daemon` service

### As a normal user

- scans only user-readable files
- does not quarantine with privileged moves
- does not attempt daemon startup
- uses per-user runtime paths when system paths are not writable

## Reading errors correctly

There are two different classes of errors:

### `find` errors

Stored in:

- `/var/lib/clamdscan-tools/state/*.find.err`
- or `~/.local/state/clamdscan-tools/state/*.find.err`

These are path traversal and filesystem discovery issues.

### scan errors

Stored in the main scan log as lines like:

```text
/path/to/file: ERROR
  msg: ERROR: Can't access file ...
```

If you see `ERROR: 259` on screen, check the main log, not `find.err`.

## Typical workflows

### Desktop user scan

```bash
clamdscan-progress ~/Descargas ~/Escritorio
clamdscan-watch
```

### Full workstation scan

```bash
sudo clamdscan-progress --system
clamdscan-watch --follow
```

### Validate exclusions after editing config

```bash
clamdscan-progress --dry-run ~/Descargas
clamdscan-watch
```
